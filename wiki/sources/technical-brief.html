Defaults

  • Current implementation direction: Tauri desktop app with Rust-owned runtime behavior while preserving feature compatibility and existing project/wiki data compatibility. The legacy Node npm package code remains as migration reference and smoke-test coverage.
  • Product category: Localhost Tooling for docs-driven agentic development.
  • HTML canonical wiki files.
  • Vanilla browser JS workspace.
  • xterm.js for terminal rendering and node-pty plus WebSocket for local PTY sessions.
  • If node-pty cannot spawn in the current runtime, the backend reports the failure and uses a pipe-backed shell fallback so the workspace remains usable while PTY compatibility is investigated.
  • Local-only server binding by default.
  • The developer's local machine, repo files, Git state, terminal sessions, credentials, and environment variables define the primary trust boundary.
  • hyperwiki launch is the primary startup command. Bare hyperwiki also launches the local workspace by default; hyperwiki dev remains the lower-level server-only command.
  • Known projects are tracked in a user-level registry at ~/.hyperwiki/projects.json; only hyperwiki-initialized repos are eligible in v1.
  • The project switcher stays hidden until more than one project is registered.
  • Agent startup is project-configured through agent.launchCommand, but workspace terminals are created lazily from the Execute control instead of opening on first load.
  • Terminal panels are scoped to the active plan root: feature-plan pages use their feature plan, MVP unit pages use their parent stage, and other app or wiki routes use separate runtime scopes.
  • Switching plans or projects detaches visible terminal panels without killing the underlying local process; explicit terminal close remains the process termination boundary.
  • Execute is a single main-checkout action. Worktree execution starts from the terminal branch rail + worktree flow, which creates and registers a checkout before sending the agent an existing-worktree handoff.
  • Active checkout identity is carried by the workspace URL shape /workspace/<project-slug>/<worktree-slug>; reloading or navigating back to that URL must preserve the worktree instead of falling back to main.
  • Portless is the default repo dev preview surface through pnpm dev, with main at https://hyperwiki.localhost and worktree URLs shaped as https://<branch-slug>.hyperwiki.localhost.
  • Repo-local worktree commands are exposed as pnpm wt:doctor, pnpm wt:create, pnpm wt:list, pnpm wt:resume, pnpm wt:open, pnpm wt:finish, and pnpm wt:prune.
  • Feature worktrees live under ../hyperwiki.worktrees/<branch-slug> and use a merge finish policy to preserve feature history.
  • The current local HTTP API exposes workspace summaries, guardrails, repo context, wiki pages, projects, worktree creation, sessions, agent prompt handoff, versioned verification loops, a versioned project contract, named agent review workflows, and a versioned MCP surface contract.
  • /api/verification derives verification loops from .hyperwiki/config.json or package scripts and merges latest runtime evidence from ignored .hyperwiki/state/verification.json.
  • /api/project-contract composes project config, Git/repo state, current plan, source briefs, verification loops, guardrails, layout, wiki page index, runtime boundaries, and copy-pasteable agent context.
  • /api/review-workflows exposes diff, architecture consistency, security, and test-gap review workflows; /api/review-workflows/run prepares or routes project-contract-aware prompts into the visible agent terminal, with results treated as runtime evidence until recorded.
  • /api/mcp-surface defines MCP-style resources and tools for project contract, current plan, source index, verification loops, guardrails, review workflows, wiki pages, review preparation, and controlled prompt submission.
  • hyperwiki mcp serves a local stdio MCP transport for read-only resources and read-only tools over Content-Length framed JSON-RPC messages.
  • It does not yet expose action-oriented MCP tools for terminal prompt submission or review workflow execution.
  • The Hyperwiki scaffold contract keeps wiki/sources.html as the source index, wiki/AGENTS.html as the visible wiki guide, and app-served /assets/wiki.css as the default generated-page stylesheet.
  • The workspace must visibly separate canonical repo truth from ignored local runtime data: wiki/ and Git are durable; .hyperwiki/state/ and .hyperwiki/sessions/ are local retention surfaces.
  • Terminal history and scrollback are runtime data unless the user intentionally exports or records them into wiki files. Export actions return runtime data to the caller and do not write canonical wiki pages automatically.
  • Hosted sync, telemetry, remote execution, and cloud-backed project memory are not default assumptions; any future hosted integration must be explicit and subordinate to local repo truth.
  • The Tauri rewrite does not use a Node sidecar as the long-term core architecture. Node modules are development/reference tooling unless deliberately removed or archived in a later cleanup.
  • Tauri desktop UI automation has platform-specific constraints, so the rewrite validation plan should combine Rust tests, compatibility smoke tests, targeted app checks, and explicit manual desktop dogfood evidence.
