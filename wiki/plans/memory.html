Memory Plan

Summary

  • Status: draft
  • Shape: product-direction plan with per-project and overall memory tracks
  • Current unit: none
  • Next action: define memory boundaries, storage locations, consent rules, and promotion paths before implementation.
  • Blockers: needs privacy and retention decisions before any persistent memory write path.
  • Validation: storage-boundary review, export/delete tests, and dogfood review across at least two registered projects.

Intent

Add memory to HyperWiki without undermining repo-local truth. Memory should help agents and humans avoid rediscovery, preserve durable preferences and decisions, and separate project-specific knowledge from cross-project operating knowledge.

Memory Types

  • Per-project memory: durable facts, preferences, decisions, verification history, and recurring context for one repository.
  • Overall memory: user-level preferences, repeated workflows, trusted commands, interface habits, and cross-project agent operating norms.
  • Runtime memory: temporary session and terminal state under ignored local storage; useful for continuity but not canonical by default.

Scope

  • Define explicit storage boundaries for project memory under the repo and overall memory under user-level HyperWiki state.
  • Define promotion flows from runtime observations to proposed memory, and from proposed memory to accepted durable memory.
  • Expose memory to agents through copy-pasteable context, local APIs, and future MCP resources.
  • Support review, export, prune, and delete operations so memory remains auditable and user-controlled.
  • Prevent accidental leakage of one project's private context into another project's agent session.

Non-goals

  • Do not silently persist terminal transcripts, prompts, secrets, or private repo details as durable memory.
  • Do not use vector storage as the first memory primitive unless a concrete retrieval problem requires it.
  • Do not treat user-level memory as canonical project truth.
  • Do not sync memory to a hosted service in this plan.

Execution Units

Unit 01 - Memory Boundary Model

  • Status: pending
  • Define where per-project memory, overall memory, and runtime memory live.
  • Define which memory is repo-visible, which is ignored local state, and which is user-level state.
  • Define consent and review rules before any durable memory write.
  • Verification: manual architecture review against existing .hyperwiki/state/, .hyperwiki/sessions/, wiki/, and ~/.hyperwiki/projects.json boundaries.

Unit 02 - Per-project Memory

  • Status: pending
  • Add a repo-local memory surface for durable project facts, preferences, decisions, recurring warnings, and verification learnings.
  • Prefer human-readable wiki-backed memory first, with structured metadata derived from or linked to it.
  • Keep project memory scoped to the selected project root and visible in the workspace contract.
  • Verification: initialize/dogfood in one project, switch to another project, and confirm memory does not cross project roots unless explicitly promoted to overall memory.

Unit 03 - Overall Memory

  • Status: pending
  • Add a user-level memory surface for preferences and operating patterns that should follow the user across HyperWiki projects.
  • Separate user preferences from project facts, and require clear promotion from project memory to overall memory.
  • Expose user-level memory in a way agents can consume without receiving unrelated project secrets.
  • Verification: dogfood across at least two projects and inspect the context provided to each agent session.

Unit 04 - Memory Review And Controls

  • Status: pending
  • Add review, accept, reject, edit, export, prune, and delete flows for proposed and durable memories.
  • Make memory writes auditable and reversible.
  • Keep controls local-first and compatible with future MCP tools.
  • Verification: smoke tests for export/delete behavior and manual review of accidental-persistence risks.

Unit 05 - Agent Memory Context

  • Status: pending
  • Include relevant per-project and overall memory in agent prompt wrappers and future MCP project contracts.
  • Label memory provenance clearly so agents know whether a fact is project-specific, user-level, or runtime-only.
  • Verification: dogfood prompt submission and inspect whether agent behavior uses memory while respecting project boundaries.
