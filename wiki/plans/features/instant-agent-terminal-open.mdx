  <PlanSummaryItem label="Status" value="completed" />

<PlanSection title="Intent">
Clicking <code>+ agent</code> or any button that starts a visible terminal-backed agent should produce a visible terminal pane immediately. If a hidden general Codex session is already warm for the active plan scope, the click should promote it where reuse is allowed; otherwise the pane should appear optimistically while the backend starts the real terminal session.
</PlanSection>

<PlanSection title="Implementation">
- Render a pending visible agent pane before awaiting <code>/api/terminal/start</code>, using a client-generated terminal id that the backend accepts.
- Preserve pending panes during session reloads until the matching real session arrives or startup fails.
- Add one hidden standby <code>purpose="general"</code> agent prewarm for the active incomplete plan scope, separate from Modify prewarm.
- Promote matching hidden general sessions when <code>+ agent</code> is clicked.
- Route Execute, New Plan, review, worktree, and import-planning handoffs through the same optimistic visible agent startup path.
- Add timing logs for click, pending pane insertion, backend start, first output, and prompt readiness.
</PlanSection>

<PlanSection title="Verification">
- <code>pnpm run check</code>
- Static smoke coverage for optimistic panes, pending preservation, and general-vs-modify prewarm separation.
</PlanSection>

<PlanSection title="Result">
Completed on 2026-06-06. Manual <code>+ agent</code> now promotes a hidden general agent when available, otherwise it inserts a pending visible pane immediately with a client-generated terminal id and reconciles that pane when the backend start returns. Follow-up extended that same optimistic pane path to Execute, New Plan, review, worktree, and import-planning handoffs. Session reloads preserve pending panes, failed optimistic starts stay dismissible, and timing milestones are logged.
</PlanSection>

<PlanSection title="2026-06-06 Readiness Follow-up">
Execute readiness now treats incomplete MCP startup counters, including compact <code>startingmcpservers(2/3)</code> terminal text, as not ready before prompt submission. Visible agent panes also keep showing the startup notice even when session metadata does not include a command string yet.
</PlanSection>

<PlanSection title="2026-06-06 Terminal Robustness Follow-up">
- Replaced the opaque prompt-ready boolean with a readiness snapshot that logs blocking states for queued follow-up input, model loading, missing prompts, incomplete MCP counts, and prompts that appear before MCP startup has completed.
- Kept the React startup notice visible until xterm receives displayable replay or live output, with limited diagnostics for chunks that produce parser-visible text but no xterm display text.
- Debounced and serialized hidden general/modify prewarm so route changes do not start a burst of background Codex sessions while visible Execute or planning handoffs are launching.
</PlanSection>

<PlanSection title="2026-06-06 Xterm Display Fallback">
When terminal output reaches the React parser but xterm does not visibly paint canvas text, the pane now shows a read-only plain transcript overlay while keeping xterm mounted for input. The display path logs xterm open, fit, write, render-check, and fallback show/hide state with geometry, row/column, canvas, and painted-pixel details for troubleshooting.
</PlanSection>

<PlanSection title="2026-06-06 Stale MCP Counter Follow-up">
Prompt readiness now treats an incomplete MCP counter as stale when a known Codex starter prompt appears after it. This avoids the full readiness timeout in cases where terminal replay still contains old <code>2/3</code> startup text but Codex is already accepting input.
</PlanSection>
