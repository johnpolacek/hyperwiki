Project Plan Creation

Plan the in-app workflow for creating a durable Hyperwiki plan inside any selected project, using an LLM-led clarification loop, the project root as the write boundary, and the project wiki as the source of truth.

Reader Goal

After 2 minutes, the reader can start implementation of project-scoped plan creation with a clear Q&A contract: the LLM keeps asking useful questions until the plan has enough clarity to write, while still avoiding endless interrogation.

Visual Concept

Local planning control panel. The feature should feel like a precise repo operation with an interview rail: scoped path preview, explicit write targets, visible open questions, confidence gates, conflict checks, and one clear post-create handoff into the new project plan.

Intent

Extend Hyperwiki's completed New Plan Workflow so plan creation works from any active project workspace, not only as a plan for Hyperwiki's own repository. The app should interview the user through an LLM-guided clarification loop, create or update files under the selected project's wiki/ directory only after the clarity gate passes, update that project's planning dashboard, and navigate the user into the newly created project plan.

Project-Scoped Flow

Scope

Area In Scope Out Of Scope
Project boundary Use the selected project or checkout root from the Rust project registry and refuse writes outside that root. Creating plans in arbitrary folders not registered as the active project.
Generated files Create compact feature plans under wiki/plans/features/<slug>.html, update wiki/plans/index.html, and append wiki/log.html when durable project context changes. Generating large MVP stage trees by default.
UI flow Expose a dense creation surface with an LLM interview rail, title, intent, plan type, generated path, conflict checks, and post-create navigation. Starting implementation automatically before the user reviews the new plan, or writing a plan while important unanswered questions remain.
Compatibility Preserve Hyperwiki scaffold conventions: lowercase wiki/sources.html, app-visible wiki/AGENTS.html, and app-served /assets/wiki.css. Applying external project-wiki filename conventions that would create duplicate wiki/Sources.html files.

Q&A Clarification Loop

Gate LLM Behavior Exit Condition
Initial intake Ask for the feature outcome, user workflow, non-goals, affected surface, validation expectation, and hard constraints not visible from repo/wiki context. The LLM can summarize the intended plan in concrete terms and name remaining unknowns.
Clarity scoring Classify open questions by impact: blocking, important, optional, or safely deferrable. Ask the next batch from the highest-impact unresolved category. No blocking questions remain, and important unknowns are answered or explicitly recorded as assumptions.
Continuous follow-up After each answer, detect contradictions, newly exposed risks, missing acceptance criteria, schema/API/UX ambiguity, and validation gaps. Ask follow-ups instead of rushing to write. The plan has clear scope, non-goals, dependencies, implementation shape, first execution unit, and verification path.
Write approval Present a compact proposed plan shape and ask for confirmation only when the remaining ambiguity would materially affect the plan. The user approves the shape, asks to skip questions, or all remaining unknowns are low-risk and documented.

Execution Units

  1. Unit 01 - Clarification contract

    Define the LLM interview state, clarity scoring, question batching, answer tracking, contradiction handling, and write gate that must pass before plan files are generated.

    Verification: Rust or fixture tests for clarity states, blocked writes, deferred unknowns, and generated prompt/context payloads.

  2. Unit 02 - Contract and routing

    Add the Rust command/API path that accepts project id, title, intent, plan type, and optional first unit, then resolves the selected project root safely.

    Verification: Rust tests for root resolution, slug conflicts, path refusal, and payload validation.

  3. Unit 03 - Wiki file generation

    Write the compact plan page, update the selected project's planning dashboard, and append a project-context log entry only when appropriate.

    Verification: fixture tests that inspect generated HTML and confirm no placeholder leakage.

  4. Unit 04 - Workspace UI

    Add the project-scoped creation surface, Q&A rail, path preview, dirty-state warning, conflict handling, clarity status, and create action from the planning workspace.

    Verification: frontend syntax check and browser or agent-browser smoke of the create modal/form states.

  5. Unit 05 - Activation and recovery

    After creation, navigate to the new plan, refresh wiki navigation, scope Execute and terminal context to the new plan, and recover cleanly after reload.

    Verification: end-to-end app smoke using a temporary project fixture and pnpm run check.

Decisions And Assumptions

  • Decision Default to compact feature plans; promote to staged plans only after the user chooses a larger planning shape.
  • Decision Creation should navigate to the new plan for review; it should not launch an agent implementation session automatically.
  • Decision The LLM should keep asking clarification questions until no blocking unknowns remain and important unknowns are answered or explicitly recorded as assumptions.
  • Assumption The first implementation should support feature, fix, cleanup, and research labels as metadata without creating different file trees.
  • Unknown Whether pasted briefs should be captured as a source brief during creation or deferred to a later import flow.
