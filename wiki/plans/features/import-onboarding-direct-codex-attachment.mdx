<h1>Import Onboarding Direct Codex Attachment</h1>
<section className="summary">
  <h2>Summary</h2>
  <ul>
    <li>Status: draft</li>
    <li>Shape: candidate architecture plan</li>
    <li>Candidate: 3 - Direct Codex app-server or daemon attachment with a minimal hyperwiki wrapper</li>
    <li>Goal: preserve Codex as the primary session engine while eliminating per-turn CLI startup, frontend-owned orchestration, and false stalled states.</li>
    <li>Next action: compare this candidate against the backend state machine, sidecar service, and broader engine candidates before implementation selection.</li>
  </ul>
</section>
<section>
  <h2>Intent</h2>
  <p>Use Codex app-server, or a future Codex daemon interface, as the long-lived agent runtime for import onboarding. hyperwiki should attach to Codex, normalize its events, and render state, but avoid building a broad new engine unless Codex proves unable to support the required user experience.</p>
  <p>This candidate is the smallest architectural move that still addresses the current failure mode: no more spawning a full CLI turn for each interaction, no more React-managed request protocol, and no more treating a quiet assistant stream as a hard stall after 25 seconds.</p>
</section>
<section>
  <h2>Architecture</h2>
  <ul>
    <li>Keep a single persistent <code>codex app-server --listen stdio://</code> process owned by the Tauri backend during app startup or first import-onboarding use.</li>
    <li>Wrap Codex app-server behind a narrow Rust <code>CodexSessionClient</code> that owns process health, JSON-RPC request IDs, thread creation, turn start, event subscription, cancellation, and restart policy.</li>
    <li>Expose import-onboarding operations from Rust to React as typed commands and events, but let Codex remain the source of the conversational thread and generation state.</li>
    <li>Use Codex app-server event types such as agent-message deltas, turn completion, errors, and tool requests as the raw stream, then normalize them into hyperwiki onboarding events.</li>
    <li>Prefer Codex-native structured output, JSON schema, or tool-request semantics when available. Fall back to strict prompt contracts and Rust-side validation when the Codex protocol cannot enforce structure.</li>
    <li>Persist only the minimal hyperwiki attachment state needed to recover: project ID, Codex thread/session ID, active turn ID, last known onboarding phase, last valid question, and durable Q&amp;A path.</li>
  </ul>
</section>
<section>
  <h2>Backend Contract</h2>
  <ul>
    <li><code>start_import_onboarding</code> ensures Codex app-server is healthy, creates or resumes the Codex thread, and starts the first onboarding turn.</li>
    <li><code>answer_import_onboarding_question</code> records the user answer, sends it into the same Codex thread, and starts the next turn.</li>
    <li><code>get_import_onboarding_status</code> returns the current attachment snapshot and any active turn state after reload.</li>
    <li><code>cancel_import_onboarding_turn</code> cancels the active Codex turn when supported, or marks the local attachment cancelled and restarts Codex before the next attempt.</li>
    <li>The backend emits typed UI events for starting, Codex healthy, turn started, assistant delta, question ready, waiting for Codex, answer recorded, plan generation, plan ready, retryable failure, fatal failure, and cancelled.</li>
  </ul>
</section>
<section>
  <h2>Flow</h2>
  <ul>
    <li>Import creates or selects a hyperwiki project, then Rust attaches to the persistent Codex app-server process.</li>
    <li>Rust creates or resumes a Codex thread seeded with compact source context and the import-onboarding protocol.</li>
    <li>Codex starts a turn to produce one source-grounded question or generate the final plan when no blocking unknowns remain.</li>
    <li>Rust streams Codex events, validates the first complete question payload, persists the attachment snapshot, and emits a question-ready event.</li>
    <li>React submits the answer to Rust. Rust writes durable Q&amp;A, sends the answer into the same Codex thread, and starts the next Codex turn.</li>
    <li>When Codex writes final plan docs, Rust validates actual files under <code>wiki/plans/</code> before marking onboarding complete.</li>
  </ul>
</section>
<section>
  <h2>UX Requirements</h2>
  <ul>
    <li>Show visible progress as soon as the backend attaches to Codex, even before assistant text arrives.</li>
    <li>First and follow-up questions should appear quickly when Codex is healthy because the app-server process and thread are already warm.</li>
    <li>Replace the 25-second no-assistant-text stall with phase-aware waiting, heartbeat detection, protocol-level turn timeouts, and explicit retry actions.</li>
    <li>Render partial assistant progress only when it improves confidence. The user-facing milestone remains the first valid question or plan-ready state.</li>
    <li>If Codex restarts, the UI should explain that the agent runtime restarted and offer a retry instead of losing the project import state.</li>
  </ul>
</section>
<section>
  <h2>Experiment Tracks</h2>
  <ul>
    <li>Test persistent <code>codex app-server</code> thread reuse against fresh <code>codex exec --json</code> turns for first-question latency and follow-up latency.</li>
    <li>Test Codex-native structured output or tool-request user input if the local protocol supports it well enough for production use.</li>
    <li>Measure whether keeping the same Codex thread improves answer continuity without making follow-up questions slower or less deterministic.</li>
    <li>Keep the implementation ready to swap in a future Codex daemon transport behind the same <code>CodexSessionClient</code>.</li>
  </ul>
</section>
<section>
  <h2>Tradeoffs</h2>
  <ul>
    <li>Benefits: smallest implementation surface, minimal packaging risk, direct use of Codex's existing session model, and likely fastest path to remove current stalls.</li>
    <li>Costs: hyperwiki remains constrained by Codex app-server protocol stability, cancellation semantics, event ordering, structured-output support, and recovery behavior.</li>
    <li>Risk: if Codex app-server is quiet or opaque during long work, this candidate may still struggle to produce the best visible UX without adding a stronger hyperwiki-owned state machine.</li>
    <li>Mitigation: normalize every Codex event into explicit UI state, add protocol watchdogs, and keep Candidate 1's backend state machine as the fallback escalation path.</li>
  </ul>
</section>
<section>
  <h2>Verification</h2>
  <ul>
    <li>Codex app-server starts once and remains attached across multiple import-onboarding turns.</li>
    <li>Fresh import produces visible progress immediately and a first valid question without frontend-owned polling.</li>
    <li>Answer submission reuses the same Codex thread and produces a follow-up question or final plan state.</li>
    <li>No false failure occurs solely because assistant text was absent for 25 seconds while the turn is otherwise alive.</li>
    <li>Malformed structured output becomes a retryable failure with raw event diagnostics available in logs.</li>
    <li>Codex restart, turn error, cancellation, and app reload leave the local attachment state recoverable.</li>
    <li>Generated plan completion is based on actual wiki files, not assistant text.</li>
    <li>Run <code>pnpm run check</code>, <code>pnpm run check:tauri</code>, and a focused Codex app-server import-onboarding smoke test.</li>
  </ul>
</section>
<section>
  <h2>Assumptions</h2>
  <ul>
    <li>Best end-user UX remains the primary criterion, but this candidate assumes Codex can provide enough streaming and session behavior without a separate hyperwiki engine.</li>
    <li>Codex-generated final plan docs remain preferred over deterministic Rust plan generation.</li>
    <li>Resume is useful but secondary to no stalls, fast first question, and fast follow-up question.</li>
    <li>Structured streaming, progressive context, human-input bridging, and artifact-contract validation remain comparison dimensions that can be layered onto this candidate.</li>
  </ul>
</section>
