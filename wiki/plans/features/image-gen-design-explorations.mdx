    <section>
      <h2>Intent</h2>
      <p>Before (or while) the Codex agent implements a unit's UI, generate candidate UI <strong>mockup images</strong> so the user can explore directions and pick one. This adds a visual ideation step ahead of implementation, then hands the chosen direction to Codex, which implements it; agent-browser then captures the real result into the existing screenshot review/gate. Image-gen = ideation, Codex = implementation, agent-browser = verification.</p>
      <p>This mirrors the per-unit screenshot infrastructure (folder storage, carousel, review dialog, awaiting/review gate, feedback queue) so most of it is a straight reuse — the only genuinely new piece is the image-generation step.</p>
    </section>
    <section>
      <h2>Storage</h2>
      <ul>
        <li>A per-unit <strong>explorations</strong> folder: <code>.hyperwiki/state/explorations/&lt;unit-path&gt;/NN-*.png</code> (gitignored runtime), mirroring <code>screenshots.rs</code>.</li>
        <li>New <code>src-tauri/src/domain/explorations.rs</code> with dir-map + list + clear, reusing the <code>screenshot_dir_for_unit</code> traversal-guarded pattern; routes mirror <code>/api/unit-screenshots</code> (GET list, GET ?path, DELETE ?path).</li>
      </ul>
    </section>
    <section>
      <h2>Generate (Codex via CLI)</h2>
      <ul>
        <li>A new command/prompt <strong>"Explore designs"</strong> sent to the Codex agent for a unit.</li>
        <li>The agent synthesizes a design brief from the unit's <code>Intent</code> / <code>Scope</code> / <code>Screen Content</code> sections, then calls the OpenAI Images API (<strong>gpt-image-2</strong>) via CLI using the project's <code>OPENAI_API_KEY</code> (already in <code>.env.local</code>), saving <strong>N</strong> candidate mockups (default ~4) into the explorations folder as ordered PNGs.</li>
        <li>Open question — confirm the exact gpt-image-2 model id, endpoint, params, and per-image cost at build time. This is the one step that is not a straight mirror of existing code; verify the API shape rather than assuming.</li>
      </ul>
    </section>
    <section>
      <h2>Review and pick</h2>
      <ul>
        <li>Reuse <code>ScreenshotCarousel</code> + a review dialog (an "explorations" variant) to step through candidates and annotate.</li>
        <li>The user <strong>picks a direction</strong> (and may add notes).</li>
      </ul>
    </section>
    <section>
      <h2>Hand to Codex</h2>
      <ul>
        <li>A "Use this direction" action injects the chosen exploration into the Execute Unit prompt — the image reference plus the user's notes, and/or a short textual description of the picked design (Codex is text-first; decide image-vs-text handoff during refinement).</li>
        <li>Codex implements → agent-browser captures the real result → the existing screenshot review and advance-gate take over.</li>
      </ul>
    </section>
    <section>
      <h2>Reuse vs new</h2>
      <p><strong>Reuse:</strong> per-unit folder storage, the carousel, the review dialog, the awaiting/review gate and feedback-queue patterns, the agent-driven <code>previewCapture</code> conventions, and the clear-then-write lifecycle.</p>
      <p><strong>New:</strong> the explorations store + routes, the "Explore designs" command, and the gpt-image-2 CLI generation step.</p>
    </section>
    <section>
      <h2>Open questions to refine</h2>
      <ul>
        <li>Confirm the gpt-image-2 model id / endpoint / params / cost; add cost guardrails (max images per run).</li>
        <li>How many variants per run (default ~4) and mockup prompt-engineering for usable UI concepts.</li>
        <li>Whether Codex consumes the chosen <em>image</em> directly or a textual brief synthesized from it.</li>
        <li>Entry points: design-first (new unit) vs redesign of an existing unit.</li>
        <li>Surface: reuse the screenshot carousel/dialog vs a dedicated "Explorations" view.</li>
        <li>Whether explorations are agent-generated only, or hyperwiki could also offer a backend path later.</li>
      </ul>
    </section>
