Reader Goal

After 2 minutes, a future agent should know which Hyperwiki scaffold conventions must be reconciled with the external Project HTML Wiki skill before editing generator code.

Intent

Hyperwiki already implements an HTML-first project memory layer, but its productized scaffold intentionally differs from the newly installed project-html-wiki skill in a few places. This plan makes those differences explicit, then updates generator code, prompts, and wiki guidance so future agents do not accidentally normalize away Hyperwiki-specific behavior.

Scope

  • Review src/init.js scaffold outputs against the skill's required project shape, naming, source index, planning, log, and automation-policy expectations.
  • Review new-project import handoff prompts in public/app.js so imported briefs ask agents to use the installed skill workflow without contradicting Hyperwiki conventions.
  • Update wiki guidance pages where needed to document intentional divergences: lowercase wiki/sources.html, generated wiki/AGENTS.html, app-served /assets/wiki.css, and Localhost Tooling runtime boundaries.
  • Record any unresolved skill-contract contradictions as explicit decisions or follow-ups instead of silently mixing conventions.

Non-Goals

  • Do not rename the existing source index to wiki/Sources.html unless a deliberate compatibility design is chosen.
  • Do not replace the app-served wiki CSS system with standalone embedded CSS across all generated pages without a product decision.
  • Do not create a repo-local maintainer skill unless the implementation decision says Hyperwiki should ship one for generated projects.
  • Do not rebootstrap this repository's existing wiki or rewrite completed plan history.

Divergence Map

Source index wiki/Sources.html wiki/sources.html generated by src/init.js and referenced by the app. Decide whether lowercase remains canonical with compatibility guidance.
Wiki agent guide wiki/AGENTS.md for repo-local wiki instructions. wiki/AGENTS.html as an app-visible HTML page plus root AGENTS.md. Document intentional HTML-first adaptation or add a Markdown companion.
Generated HTML styling Standalone HTML with embedded CSS for portable artifacts. Generated pages link to app-served /assets/wiki.css. Choose app-served scaffold as product default, standalone export mode, or hybrid.
Product boundary General HTML-first project memory layer. Localhost Tooling over local repo files, Git state, terminal sessions, credentials, and environment variables. Keep Localhost Tooling as scaffold guidance, not a generic wiki default.

Execution Units

  1. Unit 01 - Scaffold contract audit. Compare the installed skill references with src/init.js, root AGENTS.md, generated wiki pages, and project-contract outputs. Produce a small decision list before code edits.

    Verification

    Record the audited divergences in this plan or a linked implementation note before editing generator code.

  2. Unit 02 - Generator and prompt alignment. Update src/init.js and public/app.js so generated projects and imported-brief handoffs consistently describe Hyperwiki's chosen scaffold contract.

    Verification

    Run pnpm run check and the relevant scaffold/import smoke tests, including pnpm run smoke:init when generator output changes.

  3. Unit 03 - Wiki guidance sync. Update durable wiki guidance, source index, roadmap, and this plan only where implementation decisions change future-agent behavior.

    Verification

    Use agent-browser or browser review to inspect the updated wiki surface, then record any durable validation result in wiki/log.html.

Acceptance

  • Generated Hyperwiki projects have one documented scaffold contract for source index naming, wiki agent guidance, styling portability, and Localhost Tooling boundaries.
  • Import handoff prompts tell agents when to apply the installed project-html-wiki workflow and when to preserve Hyperwiki product conventions.
  • Future agents can read this plan and avoid creating duplicate Sources.html, duplicate wiki guidance files, or unmanaged scaffold rewrites.
  • Checks and visual review confirm that scaffold changes do not break existing wiki navigation or new project creation flow.
