New Plan Workflow

Summary

  • Status: active
  • Shape: compact feature plan
  • Current unit: design the + new plan workflow
  • Next action: implement an in-app plan creation flow that turns a rough intent into a durable wiki plan and current workspace target.
  • Validation: pnpm run check, browser smoke coverage for creation and reload, and an agent-browser interaction pass.

Intent

Add a + new plan workflow for starting product work from inside the existing planning surface. The flow should create canonical wiki plan files, keep the left topbar project identity stable, and make the new plan the active workspace context without requiring manual file authoring.

User Flow

  1. The user clicks + new plan from the Planning Dashboard or plan sidebar header.
  2. Hyperwiki opens a compact creation surface that asks for the plan title, short intent, and whether the work is a feature, fix, cleanup, or research spike.
  3. The app previews the generated slug, wiki path, first unit title, and the execution target that will become active after creation.
  4. On create, Hyperwiki writes the plan page, updates wiki/plans/index.html, records the action in wiki/log.html, and navigates to the new plan.
  5. The new plan becomes the current plan for Execute, terminal scoping, Up Next, and worktree creation.

UI Requirements

  • Place + new plan where planning work begins, not in the global Projects menu.
  • Use a restrained form or popover that matches the dense Hyperwiki workspace, with no marketing-style empty state or decorative cards.
  • Show generated file path and slug before writing so the user can catch naming mistakes.
  • After creation, keep the topbar unchanged and change only the wiki route, sidebar current state, command bar context, and terminal scope.
  • Use overflow-safe labels in the sidebar and breadcrumb because generated plan titles can be long.

Data And Files

  • Feature plans should default to wiki/plans/features/<slug>.html.
  • The generated page should include Summary, Intent, User Flow or Implementation, Verification, and Closeout sections.
  • The Planning Dashboard should list the new plan as active until it is closed out.
  • The project log should receive a concise planning entry when the plan is created.
  • All writes are repo-visible wiki changes; runtime state should only remember UI draft form contents if needed.

Edge Cases

  • If the slug already exists, show an inline conflict with a suggested alternate slug.
  • If the repo has uncommitted wiki changes, still allow creation but show a visible warning that the new plan will be added to the current worktree.
  • If creation partially succeeds, surface exactly which file write failed and do not mark the plan current until the plan page exists.
  • If the user reloads after creation, the active plan should be derived from the wiki plan status and URL, not from transient browser state.

Open Decisions

  • Whether + new plan should create one initial unit page or keep compact feature plans as a single page until complexity demands units.
  • Whether creation should optionally start an agent immediately, or only create and navigate to the plan so the user can review before Execute.
  • Whether the creation flow should support importing a pasted brief as source evidence in the same pass.
