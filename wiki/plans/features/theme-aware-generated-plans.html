Theme-Aware Generated Plans

Make newly generated HTML wiki plans keep their rich per-plan layouts while inheriting Hyperwiki's current wiki theme through shared CSS variables and app-served theme tokens.

Reader Goal

After 2 minutes, the reader can start implementation by auditing the theme token path, then updating generated plan templates so new plan pages are visually rich but theme-responsive.

Decision

Preserve rich per-plan layouts. Require local plan CSS to alias color, font, border, code, and surface roles to shared --docs-* variables instead of hardcoded palettes.

Migration Boundary

Leave existing plan files as they are. This feature governs newly generated plans and the generation contracts that future agents or in-app workflows use.

Reliability Preference

Prefer app-served /assets/wiki.css and runtime CSS variable propagation. Use embedded fallback variables only as defensive defaults, not as a second theme system.

Intent

Hyperwiki's generated plan pages should feel purpose-built, not flattened into a generic document, while still adapting when the user changes Hyperwiki's theme in Settings. The implementation should align the app runtime, wiki CSS, in-app plan generation, and project-html-wiki guidance around one contract: rich artifacts may define page-local layout primitives, but semantic colors and fonts must flow from the wiki theme variables.

Theme Propagation Flow

Scope And Non-Goals

AreaIn ScopeOut Of Scope
Generated plans New plan pages produced by Hyperwiki workflows or agent guidance should use shared theme roles for colors and fonts. Bulk rewriting existing plan pages or forcing all plans into one template.
Theme runtime Ensure wiki iframe pages can receive current Settings theme values reliably through app-served CSS variables. Adding a separate per-project theme editor or a second theme persistence model.
Guidance Review whether app prompts, local templates, and project-html-wiki guidance need explicit generated-plan theme rules. Changing the external skill without review if the app already owns the generation contract.
Sidebar This plan appears through the standard active feature plan path and dashboard link. A new sidebar section or custom navigation model for generated plans.

Execution Units

  1. Unit 01 - Theme contract audit

    Trace the current Settings theme model, applyTheme, wiki iframe styling, /assets/wiki.css, and any generated plan templates or prompts. Document the exact CSS variable contract generated plans should consume.

    Verification: Static inspection plus targeted checks proving current wiki pages receive theme variables when Settings changes.

  2. Unit 02 - Generation contract update

    Update the in-app plan generation prompts/templates and, if needed after review, repo-local or project-html-wiki guidance so future rich plan artifacts alias local style tokens to shared --docs-* variables.

    Verification: Generate or fixture-test a sample plan and assert it avoids hardcoded foreground/background palettes for semantic roles.

  3. Unit 03 - Visual and sidebar validation

    Validate a newly generated rich plan under at least two Settings themes, confirm readable contrast and layout integrity, and confirm the normal plan sidebar discovers the new feature plan path.

    Verification: pnpm run check, pnpm run smoke:tauri-static-assets, and app/browser visual QA for theme switching.

Risks And Open Decisions

TopicRiskPlan Response
Iframe theme propagation App theme variables may be applied to the shell but not consistently to wiki iframe documents. Audit and test the runtime path before changing generation rules.
Rich layout contrast Color-mixed local accents may become low contrast under unusual custom themes. Define safe token roles and add visual QA across light/dark or custom presets.
Skill scope Theme rules may belong in Hyperwiki's app prompts, the external project-html-wiki skill, or both. Review actual generation ownership in Unit 01 before editing any skill guidance.
