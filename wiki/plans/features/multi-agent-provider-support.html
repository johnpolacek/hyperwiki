Multi-Agent Provider Support

Summary

  • Status: active
  • Shape: compact feature plan
  • Current unit: Unit 01 - Provider-Aware Agent Terminals
  • Next action: implement provider-aware Codex and Claude agent terminal support while keeping Codex as the default.
  • Validation: pnpm run check, targeted smoke coverage, and browser verification of provider controls.

Intent

Support Claude alongside Codex without weakening Hyperwiki's visible terminal handoff model. Codex remains the default provider for Execute and new handoffs, while users can start Claude sessions and run Codex and Claude side by side in the same workspace.

Scope

  • Add provider-aware agent configuration with Codex as the default provider and Claude as an additional local CLI provider.
  • Preserve legacy agent.launchCommand by treating it as the Codex provider command when newer provider configuration is absent.
  • Allow multiple visible agent terminals with provider identity in session metadata and terminal chrome.
  • Route Execute, dashboard, plan, review, and MCP prompt handoffs to the default Codex provider unless a specific provider or session is selected.
  • Keep provider-specific behavior isolated behind adapters: Codex keeps its readiness detection and thinking-effort flag handling; Claude uses the interactive claude CLI path and never receives Codex-specific flags.

Non-Goals

  • Do not remove Codex defaults or require users to choose a provider before Execute.
  • Do not add hosted Claude, Anthropic SDK, or headless claude -p workflows in the first pass.
  • Do not install Claude, create credentials, or manage external provider authentication.
  • Do not change the Localhost Tooling boundary: provider CLIs run as local visible terminal processes.

Unit 01 - Provider-Aware Agent Terminals

  • Status: active
  • Normalize agent config to expose defaultProvider plus provider command entries for Codex and Claude.
  • Infer provider metadata for existing terminal layout panels so older projects keep working.
  • Update terminal creation and prompt submission to target the default provider, an explicit provider, or an explicit active session.
  • Replace Codex-only prompt readiness and paste handling with provider adapters.
  • Update the terminal toolbar to create Codex or Claude agent terminals and show provider labels clearly when mixed sessions are active.
  • Update project contract and MCP surface metadata so downstream agents can see the default provider and available provider commands.

Verification

  • pnpm run check passes.
  • Legacy agent.launchCommand: "codex --yolo" still creates a Codex default provider and launches existing agent flows.
  • Fresh provider config defaults Execute and handoffs to Codex while exposing Claude as an available provider.
  • Creating Codex and Claude terminals results in separate visible agent sessions with clear provider labels.
  • Prompt submission can target Claude without waiting for the Codex banner readiness condition.
  • Codex thinking-effort overrides are appended only to Codex commands and never to Claude commands.
  • Browser verification covers the provider creation controls, mixed terminal labels, Execute handoff, and plan prompt handoff.

Assumptions

  • Default to Codex means the default provider for new handoffs and Execute actions, not automatic startup of Codex terminals on workspace load.
  • Allow a mix means simultaneous visible Codex and Claude terminal sessions in one workspace.
  • Claude support starts with the local interactive claude CLI command. Official Claude Code docs identify claude as interactive mode and claude -p as print mode.
