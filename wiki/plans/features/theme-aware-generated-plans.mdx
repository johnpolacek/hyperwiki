    <section className="grid" aria-label="Planning decisions">
      <article className="decision">
        <strong>Decision</strong>
        <p>Preserve rich per-plan layouts. Require local plan CSS to alias color, font, border, code, and surface roles to shared <code>--docs-*</code> variables instead of hardcoded palettes.</p>
      </article>
      <article className="decision">
        <strong>Migration Boundary</strong>
        <p>Leave existing plan files as they are. This feature governs newly generated plans and the generation contracts that future agents or in-app workflows use.</p>
      </article>
      <article className="decision">
        <strong>Reliability Preference</strong>
        <p>Prefer app-served <code>/assets/wiki.css</code> and runtime CSS variable propagation. Use embedded fallback variables only as defensive defaults, not as a second theme system.</p>
      </article>
    </section>

    <section className="panel" aria-labelledby="intent">
      <h2 id="intent">Intent</h2>
      <p>hyperwiki's generated plan pages should feel purpose-built, not flattened into a generic document, while still adapting when the user changes hyperwiki's theme in Settings. The implementation should align the app runtime, wiki CSS, in-app plan generation, and hyperwiki guidance around one contract: rich artifacts may define page-local layout primitives, but semantic colors and fonts must flow from the wiki theme variables.</p>
    </section>
