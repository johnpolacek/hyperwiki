## How this works

A project advances through six phases in order. The **active phase** is the first phase whose gate is not yet cleared; only its handoff is enabled. A master **orchestrator** (Ask the Orchestrator) reads this lifecycle, names the active phase and its blocking gate, and recommends the next handoff — it never executes phase work itself. Each phase hands off to a specialized sub-agent that loads the right repo-local skills.

## Gate algorithm

The active phase is the first phase, in order, whose gate is not cleared:

- `gate: childPlan` — cleared when the phase page is `complete` and its linked `childPlan` is `complete`.
- `gate: manual` — cleared when the phase page is marked `complete` (used for the Phase 4 architecture decision).
- `gate: import-validated` — cleared when `wiki/sources/prd.mdx` has user stories, or the imported-project plan artifacts validate.

<Aside>
Phase 1 has no `childPlan`: it establishes purpose and produces `wiki/sources/prd.mdx`. The MVP **build** sub-plan is owned by Phase 6 (`childPlan: /wiki/plans/mvp/`), keeping "establish purpose" distinct from "build the views".
</Aside>
