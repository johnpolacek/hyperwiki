## Stage Goal

The app can derive lifecycle phase state (which phase is active, whether each gate is cleared) from `wiki/plans/lifecycle/` pages, entirely in new code, without altering any existing exported function in `wiki-pages.ts`.

<Dependencies>

- None. This stage is additive and ships before any wiki content or UI exists.

</Dependencies>

### Unit 02 — lifecycle.ts derivation

<Scope>

- New file `src/lib/lifecycle.ts`. Reuses `childPlanPages`, `pageStatus`, `isCompletedPage` from `wiki-pages.ts` without editing them.

</Scope>

<ImplementationNotes>

- `LIFECYCLE_PHASES`: ordered descriptor array — `{ phaseId, label, phaseOrder, archetype: "planning" | "execute", skills: string[], childPlan?: string, gate: "childPlan" | "manual" | "import-validated" }` matching the [index](./index.mdx) phase table.
- Pure helpers: `isLifecyclePlanPage(page)`, `lifecyclePhases(pages)` (returns the 6 phase pages joined to their descriptors, in `phaseOrder`), `phaseGateCleared(phase, pages)` (implements the three-case gate algorithm), `activeLifecyclePhase(pages)` (first phase whose gate is not cleared).
- Read `phaseId`/`phaseOrder`/`childPlan`/`gate` from page frontmatter; fall back to the descriptor when a page omits them.

</ImplementationNotes>

<Verification title="Verification">
  <CommandBlock>pnpm test lifecycle</CommandBlock>
  - Unit tests for `phaseGateCleared` across all three gate kinds and for `activeLifecyclePhase` ordering, mirroring the existing `wiki-pages` test style in `tests/`.
</Verification>
