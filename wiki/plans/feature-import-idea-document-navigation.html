Import Idea Document Navigation

Summary

  • Status: active
  • Shape: compact corrective feature plan
  • Current unit: implement the deterministic create-and-navigate idea import flow
  • Next action: add the local create API, wire Dashboard idea submission to it, and update smoke coverage.
  • Blockers: none
  • Validation: pnpm run check, pnpm run smoke:ideas, pnpm run smoke:browser, plus a browser visual check at https://feature-import-idea-document-navigation.hyperwiki.localhost.

Problem

The Dashboard idea import currently starts an agent with instructions to create wiki/ideas/<slug>.html. When importing an idea document, the expected product behavior is for hyperwiki itself to create a new idea document and navigate to it.

This plan changes idea import from an agent handoff into deterministic app behavior while preserving project import as an agent-assisted workflow unless explicitly changed later.

Intended Outcome

  • Selecting or entering an idea document from the Dashboard creates a new HTML idea page under wiki/ideas/.
  • The app navigates to the newly-created idea page after creation so the user can review and continue from the canonical document.
  • The created page is visible in the Ideas list and keeps the existing idea promotion action.
  • Failures are shown as Dashboard status without starting an unrelated agent session.

Confirmed Decisions

  • Imported HTML should be kept nearly as-is inside the wiki layout.
  • Imported Markdown should be converted to HTML for the created idea page.
  • Slug collisions should auto-suffix the new page path instead of overwriting or blocking.
  • Successful creation should navigate directly and automatically to /wiki/ideas/<slug>.html in the current workspace frame.
  • The form can clear after success; navigation away from the Dashboard is the visible outcome.
  • Typed manual ideas and file imports should both use the same create-and-navigate flow when the form is valid.

Likely Implementation

  • Add a local HTTP endpoint such as POST /api/ideas/create in src/server.js.
  • Accept title, imported content, and document type from the Dashboard; validate required fields and keep writes constrained to wiki/ideas/.
  • Generate a unique slug if wiki/ideas/<slug>.html already exists.
  • Render imported Markdown into HTML and preserve imported HTML content nearly as provided inside the existing wiki HTML layout.
  • Replace handOffIdeaMarkdown in public/app.js with an API call, list refresh, and navigation to the returned idea URL.
  • Update tests/ideas-smoke.mjs and tests/browser-smoke.mjs so coverage expects page creation and navigation rather than agent handoff text.

Non-Goals

  • No change to idea promotion into full projects.
  • No change to Dashboard project import unless a later decision says project imports should also become deterministic.
  • No new database or hidden idea store; ideas remain repo-visible HTML files under wiki/ideas/.
  • No broad redesign of Dashboard layout or idea page styling.

Acceptance Criteria

  • Creating an idea from typed content writes a unique HTML page under wiki/ideas/ and navigates to it.
  • Creating an idea from imported Markdown or HTML infers the title, writes the page, and navigates to it.
  • Imported HTML content is preserved nearly as-is inside the idea page layout; imported Markdown is converted to HTML.
  • Duplicate titles produce an auto-suffixed slug rather than overwriting an existing idea.
  • The API rejects paths or titles that would escape wiki/ideas/.
  • Existing idea listing and promotion behavior continue to work for created idea pages.
  • Automated smoke coverage proves the create-and-navigate behavior in both server-level and browser-level flows.
