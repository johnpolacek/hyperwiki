Product Soul Plan

Summary

  • Status: draft
  • Shape: product-direction plan
  • Current unit: none
  • Next action: define the durable product personality principles that should guide copy, defaults, interface behavior, and agent interactions.
  • Blockers: needs user taste calibration before implementation.
  • Validation: design review, dogfood review, and consistency checks across generated wiki pages, workspace UI, CLI output, and agent prompts.

Intent

Give HyperWiki a coherent soul: a recognizable product character that makes the local agent workspace feel purposeful, calm, durable, and alive without turning into a mascot, gimmick, or marketing surface.

Working Definition

Soul is the product's durable taste layer. It shows up in what HyperWiki chooses to make visible, what it refuses to hide, how it names actions, how it reacts to uncertainty, and how it helps humans and agents preserve understanding.

Scope

  • Define product personality principles for language, visual tone, interaction defaults, empty states, status messages, and agent prompt framing.
  • Create a small set of reusable voice and behavior rules for generated wiki pages, CLI output, workspace chrome, guardrails, and review workflows.
  • Make the product feel locally grounded, trustworthy, and intentional while preserving the existing utilitarian workspace posture.
  • Clarify what HyperWiki should never become: noisy, anthropomorphic, patronizing, decorative-first, or a hidden source of truth.

Non-goals

  • Do not add a mascot, novelty persona, or fictional character as the soul.
  • Do not create a public marketing page as part of this plan.
  • Do not make the interface sentimental at the expense of clarity or density.
  • Do not change agent autonomy, memory, or verification behavior here; those belong to dedicated plans.
