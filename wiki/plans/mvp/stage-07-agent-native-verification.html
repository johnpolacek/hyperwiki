Stage 07 - Agent-native Verification

Summary

  • Status: active
  • Shape: single MVP stage with multiple execution units
  • Current unit: Unit 01 - Verification Loop Model
  • Next action: define the repo-local verification loop data model and API contract. Treat Units 05, 06, and 07 as completed enabling workspace work, not as completion of the core verification/contract/MCP layer.
  • Blockers: none
  • Validation: pnpm run check, targeted smoke coverage for new APIs and browser-visible workflow state, and manual dogfood from an agent session.

Intent

Turn hyperwiki from an agent-visible workspace into an agent-native operating layer: repo-local context, structured tool surfaces, explicit verification loops, auditable agent review workflows, and machine-readable project contracts.

Source Context

  • Karpathy's Software 3.0 framing: define the context, tools, feedback loop, and guardrails; let agents work while preserving human understanding.
  • Existing hyperwiki direction: canonical HTML wiki files, local-only runtime state, visible guardrails, agent-first launch, and deferred structured MCP integration.
  • Product decision: keep canonical project truth in repo-visible files and Git; keep terminal/session/runtime data local unless intentionally exported or recorded.

Scope

  • Promote verification loops to a first-class product primitive across checks, tests, smoke scripts, screenshots, CI status, and manual review outcomes.
  • Define structured MCP/tool surfaces as the product thesis, not only a later technical integration.
  • Add explicit agent review workflows for security, architecture, test gaps, and diff review.
  • Preserve the repo-local wiki and Git as durable truth, avoiding hidden app state as the source of project knowledge.
  • Add machine-readable project contracts for facts, current plan state, verification commands, permission boundaries, and source index metadata.

Implementation State From Code Review

  • Implemented: Dashboard is a first-class workspace page for ideas and projects, with markdown import/create flows that start an agent handoff instead of writing generated pages directly.
  • Implemented: the wiki command bar exposes breadcrumbs, Modify, Execute, preview links, and completed-plan behavior; completed plan pages hide Modify/Execute and show a Completed badge.
  • Implemented: Execute starts terminals lazily, supports main and worktree targets, posts scoped instructions into an agent terminal, and shows Portless preview URLs for main or worktree execution.
  • Implemented: sidebar plan navigation keeps stages chronological, marks current/completed rows, pins the Project group to the bottom, and keeps completed plans under the archive group.
  • Implemented: /api/workspace exposes a workspace summary with plan status, source briefs, verification command labels, and layout configuration; /api/guardrails exposes local-only and runtime-boundary facts.
  • Not implemented yet: a first-class verification loop data model with recorded runs, evidence, status, triggers, and API contract.
  • Not implemented yet: a stable machine-readable project contract endpoint or file that composes project facts, plan state, source index, verification loops, permissions, and runtime boundaries.
  • Not implemented yet: named agent review workflows and MCP resources/tools beyond the current local HTTP workspace APIs and terminal prompt handoff.

Non-goals

  • Do not replace the HTML wiki with a database-backed knowledge silo.
  • Do not auto-commit, auto-push, or auto-resolve reviews from agent output without explicit user policy.
  • Do not build a full remote collaboration or hosted CI product in this stage.
  • Do not make MCP integration depend on one agent vendor's private protocol when a local CLI/API contract can remain stable.
