Stage 07 - Agent-native Verification

Summary

  • Status: active
  • Shape: single MVP stage with multiple execution units
  • Current unit: Unit 01 - Verification Loop Model
  • Next action: define the repo-local verification loop data model and API contract before implementing UI or MCP surfaces.
  • Blockers: none
  • Validation: pnpm run check, targeted smoke coverage for new APIs and browser-visible workflow state, and manual dogfood from an agent session.

Intent

Turn HyperWiki from an agent-visible workspace into an agent-native operating layer: repo-local context, structured tool surfaces, explicit verification loops, auditable agent review workflows, and machine-readable project contracts.

Source Context

  • Karpathy's Software 3.0 framing: define the context, tools, feedback loop, and guardrails; let agents work while preserving human understanding.
  • Existing HyperWiki direction: canonical HTML wiki files, local-only runtime state, visible guardrails, agent-first launch, and deferred structured MCP integration.
  • Product decision: keep canonical project truth in repo-visible files and Git; keep terminal/session/runtime data local unless intentionally exported or recorded.

Scope

  • Promote verification loops to a first-class product primitive across checks, tests, smoke scripts, screenshots, CI status, and manual review outcomes.
  • Define structured MCP/tool surfaces as the product thesis, not only a later technical integration.
  • Add explicit agent review workflows for security, architecture, test gaps, and diff review.
  • Preserve the repo-local wiki and Git as durable truth, avoiding hidden app state as the source of project knowledge.
  • Add machine-readable project contracts for facts, current plan state, verification commands, permission boundaries, and source index metadata.

Non-goals

  • Do not replace the HTML wiki with a database-backed knowledge silo.
  • Do not auto-commit, auto-push, or auto-resolve reviews from agent output without explicit user policy.
  • Do not build a full remote collaboration or hosted CI product in this stage.
  • Do not make MCP integration depend on one agent vendor's private protocol when a local CLI/API contract can remain stable.

Execution Units

Unit 01 - Verification Loop Model

  • Status: active
  • Define the repo-local shape for verification loops: command, scope, trigger, status, last run, evidence, and whether the result is automated or manual.
  • Expose verification loops through a local API that can be consumed by the browser workspace and future MCP tools.
  • Keep recorded verification state separate from canonical claims unless a human or agent intentionally records the outcome into the wiki.
  • Verification: pnpm run check plus targeted API smoke coverage for reading configured verification loops.

Unit 02 - Machine-readable Project Contract

  • Status: pending
  • Add a stable project contract endpoint or file that summarizes project facts, current plan state, source index metadata, verification commands, permission boundaries, and runtime-state boundaries.
  • Prefer deriving contract values from canonical wiki/config/Git state over duplicating maintained facts.
  • Make the contract copy-pasteable into agents and directly useful as MCP resource payloads.
  • Verification: contract smoke test plus manual inspection against wiki/index.html, wiki/plans/index.html, wiki/sources.html, and .hyperwiki/config.json.

Unit 03 - Agent Review Workflows

  • Status: pending
  • Add named review workflows for diff review, architecture consistency, security review, and test-gap review.
  • Route review prompts into the visible agent terminal with project contract and current plan context.
  • Represent review results as runtime evidence until intentionally recorded in the wiki or Git.
  • Verification: browser smoke coverage for workflow discovery and manual dogfood against an active agent session.

Unit 04 - MCP Surface Definition

  • Status: pending
  • Define the first MCP resources and tools around project contract, current plan, source index, verification loops, guardrails, and controlled prompt submission.
  • Keep MCP behavior local-first and aligned with existing permission boundaries.
  • Document the stable contract before adding broad implementation surface.
  • Verification: local contract tests or smoke coverage for MCP-adjacent handlers, plus manual review of permission boundaries.

Unit 05 - Workspace Framing Update

  • Status: pending
  • Update product copy and workspace labels from a terminal/wiki workspace toward a repo-local agent operating layer.
  • Keep the UI utilitarian: surface current context, verification loops, guardrails, and review workflows without creating a marketing page.
  • Verification: browser smoke coverage for visible current-plan and verification state, plus manual responsive review.
