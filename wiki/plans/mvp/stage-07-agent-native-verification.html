Stage 07 - Agent-native Verification

Summary

  • Status: complete
  • Shape: single MVP stage with multiple execution units
  • Current unit: complete
  • Next action: use the Stage 07 APIs and local stdio MCP transport while planning the next implementation stage. Completed workspace-enabler work has been split into Agent Workspace Enablers so this stage stays focused on the core verification/contract/MCP layer.
  • Blockers: none
  • Validation: pnpm run check, targeted smoke coverage for new APIs and browser-visible workflow state, and manual dogfood from an agent session.

Intent

Turn hyperwiki from an agent-visible workspace into Localhost Tooling for agent-native development: repo-local context, structured tool surfaces, explicit verification loops, auditable agent review workflows, and machine-readable project contracts inside the developer's own trust boundary.

Source Context

  • Karpathy's Software 3.0 framing: define the context, tools, feedback loop, and guardrails; let agents work while preserving human understanding.
  • Existing hyperwiki direction: canonical HTML wiki files, local-only runtime state, visible guardrails, agent-first launch, and deferred structured MCP integration.
  • 2026-05-14 product decision: hyperwiki should fully claim and identify as Localhost Tooling.
  • Product decision: keep canonical project truth in repo-visible files and Git; keep terminal/session/runtime data local unless intentionally exported or recorded.

Scope

  • Promote verification loops to a first-class product primitive across checks, tests, smoke scripts, screenshots, CI status, and manual review outcomes.
  • Define structured MCP/tool surfaces as the product thesis, not only a later technical integration.
  • Make the Localhost Tooling trust boundary explicit in project contracts, guardrails, verification loops, and agent handoffs.
  • Add explicit agent review workflows for security, architecture, test gaps, and diff review.
  • Preserve the repo-local wiki and Git as durable truth, avoiding hidden app state as the source of project knowledge.
  • Add machine-readable project contracts for facts, current plan state, verification commands, permission boundaries, and source index metadata.

Implementation State From Code Review

  • Implemented in a separate completed plan: Dashboard handoffs, command bar affordances, Execute target workflow, sidebar Project pinning, and completed-plan behavior are preserved in Agent Workspace Enablers.
  • Implemented: /api/workspace exposes a workspace summary with plan status, source briefs, verification loops, and layout configuration; /api/guardrails exposes local-only and runtime-boundary facts.
  • Implemented: /api/verification exposes versioned verification loops with status, triggers, last run, evidence, manual/automated kind, and runtime evidence boundaries.
  • Implemented: /api/project-contract exposes a stable machine-readable project contract that composes project facts, plan state, source index, verification loops, guardrails, layout, wiki pages, and runtime boundaries.
  • Implemented: /api/review-workflows and /api/review-workflows/run expose named agent review workflows for diff, architecture consistency, security, and test-gap review, with review findings kept as runtime evidence until recorded.
  • Implemented: /api/mcp-surface defines stable MCP-style resources and tools for project contract, current plan, source index, verification loops, guardrails, review workflows, and controlled prompt submission.
  • Implemented: hyperwiki mcp serves read-only Stage 07 resources and tools over local stdio MCP transport.
  • Not implemented yet: action-oriented MCP tools for terminal prompt submission or review workflow execution.

Non-goals

  • Do not replace the HTML wiki with a database-backed knowledge silo.
  • Do not auto-commit, auto-push, or auto-resolve reviews from agent output without explicit user policy.
  • Do not build a full remote collaboration or hosted CI product in this stage.
  • Do not make hosted sync, remote execution, telemetry, or cloud-backed memory default requirements.
  • Do not make MCP integration depend on one agent vendor's private protocol when a local CLI/API contract can remain stable.
