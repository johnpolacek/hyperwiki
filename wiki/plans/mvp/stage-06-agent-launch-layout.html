Stage 06 - Agent-first Launch Layout

Summary

  • Status: complete
  • Current unit: none
  • Next action: dogfood reset plus launch from a cleared registry, then define structured MCP integration.
  • Blockers: none
  • Validation: pnpm run check, pnpm run smoke:launch, pnpm run smoke:reset, pnpm run smoke:browser, pnpm run smoke:init, pnpm run smoke:sessions, pnpm run smoke:pty.

Intent

Make HyperWiki launch into the actual working set: an agent first when configured, a dev command when present, and a single CLI shell.

Execution Units

Unit 01 - Agent Launch Setting

  • Status: complete
  • Store the per-project setting at agent.launchCommand.
  • Fresh initialization leaves the setting empty unless the user passes --agent-launch-command.
  • This repository uses codex --yolo for dogfooding.
  • Implemented as agent.launchCommand in .hyperwiki/config.json.
  • Corrected after dogfooding: do not inherit another project's agent command into fresh projects.

Unit 02 - Three-panel Startup Contract

  • Status: complete
  • Panel order: agent first when configured, dev second when a dev script exists, CLI shell always.
  • Do not show empty agent or dev panels.
  • Remove checks and git from generated default layouts.
  • Implemented with a smoke-test dry-run path for agent commands.
  • Corrected after dogfooding: launch sessions are visible as separate terminal panes instead of being hidden behind tabs.
  • Corrected after dogfooding: the HyperWiki repo omits its dev panel because npx hyperwiki already starts the same local server.

Unit 04 - Workspace Chrome Reduction

  • Status: complete
  • Hide standalone wiki page chrome when wiki pages are embedded inside the workspace.
  • Replace duplicated sidebar summaries with a compact next-action status.
  • Move terminal identity into each visible terminal pane header.

Unit 05 - Agent Plan Prompt

  • Status: complete
  • Add a sidebar prompt when an agent session is configured and active.
  • Submit prompts into the visible agent terminal instead of creating a hidden chat transcript.
  • Include project, repo root, and current wiki page context in the submitted instruction.

Unit 03 - Repeatable Fresh-launch Reset

  • Status: complete
  • Add a safe command for dogfooding first-launch behavior repeatedly on the same computer.
  • Clear the user-level project registry and ignored per-project runtime state.
  • Do not remove canonical project files such as wiki/ or .hyperwiki/config.json.
  • Support --dry-run so reset scope can be inspected before deletion.
  • Implemented as npx hyperwiki reset with npx hyperwiki reset --dry-run.
