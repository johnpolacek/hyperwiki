Stage 04 - One-command Local Launch

Summary

  • Status: active
  • Current unit: Unit 01 - One-command Local Launch
  • Next action: implement and verify a launch command that starts the local server, opens the browser workspace, and prepares the default wterm session.
  • Blockers: none
  • Validation: pnpm run check, pnpm run smoke:browser, launch smoke covering server readiness, browser open intent, and terminal session creation.

Intent

Make HyperWiki obvious to start. From a fresh dogfood session, one command should bring up the local wiki/ADE workspace without requiring the user to remember separate server, browser, and terminal steps.

Dogfood Finding

The first-use friction is launch ambiguity: HyperWiki is essentially a local LLM wiki site and ADE in one, but the current workflow does not clearly answer how to launch the whole environment.

Execution Units

Unit 01 - Launch Command Contract

  • Status: active
  • Define the public command surface for starting HyperWiki as a complete local workspace. Favor the existing CLI and package scripts over introducing a separate launcher.
  • The command should start the local server, wait until the workspace is reachable, open the browser to the workspace URL, and ensure the default wterm operating session is created or visible.
  • Keep the command local-first and repo-root aware. It must preserve ignored runtime state under .hyperwiki/ and avoid writing terminal runtime data into canonical wiki files.
  • Acceptance: a user can run one documented command from the repository root and immediately land in a browser workspace with the default terminal workflow available.

Unit 02 - Launch Reliability And Ergonomics

  • Status: pending
  • Handle common local startup cases: port selection or conflict reporting, server readiness, browser-open failures, clean shutdown, and repeated invocation while a server is already running.
  • Document the command in the development workflow and generated project guidance so new users can discover it before reading source code.
  • Acceptance: repeated dogfood launches are predictable, failures are actionable, and the command output includes the workspace URL.

Unit 03 - Smoke Coverage

  • Status: pending
  • Add automated coverage for the launch path without depending on a real desktop browser in CI. Browser-open behavior can be asserted through an injectable opener or dry-run mode.
  • Verify the launched workspace exposes the expected server APIs and default terminal/session state.
  • Acceptance: launch behavior is covered by package scripts and can be validated during dogfood work before code commits.

Non-goals

  • Do not start structured MCP integration in this stage.
  • Do not introduce remote hosting, auth, or multi-user behavior.
  • Do not make runtime terminal transcripts canonical project knowledge by default.

Verification

Record launch validation in log.html before marking this stage complete.
