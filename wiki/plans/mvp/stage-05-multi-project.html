Stage 05 - Multi-project Workspace

Summary

  • Status: complete
  • Current unit: none
  • Next action: define structured MCP integration after multi-project dogfooding.
  • Blockers: none
  • Validation: pnpm run check, pnpm run smoke:launch, pnpm run smoke:browser, pnpm run smoke:init, pnpm run smoke:sessions, pnpm run smoke:pty.

Intent

Let HyperWiki manage multiple local projects while preserving the existing single-project feel until more than one project exists.

Execution Units

Unit 01 - Registry and Launch Foundation

  • Status: complete
  • Add a user-level registry at ~/.hyperwiki/projects.json.
  • npx hyperwiki from a HyperWiki-initialized repo auto-registers that repo and opens it as the active project.
  • Only repos with wiki/ and .hyperwiki/config.json are eligible.
  • Project names come from config projectName, then package name, then folder name.
  • Implemented with a user-level registry that can be isolated in tests through HYPERWIKI_HOME.

Unit 02 - Project-scoped Server Runtime

  • Status: complete
  • Add project-aware APIs and wiki routes that resolve roots through the registry.
  • Scope terminal WebSocket sessions and retained metadata to the selected project root.
  • Keep missing projects listed but disabled.
  • Implemented project-scoped API reads, wiki routes, and PTY session routing.

Unit 03 - Multi-project UI

  • Status: complete
  • Hide project switching UI when only one project is registered.
  • Show a collapsible project sidebar when two or more projects exist.
  • Switch projects in the same tab and reload wiki, repo, summary, guardrails, layout, and sessions.
  • Implemented and covered by launch smoke for hidden single-project UI and visible two-project switching.

Non-goals

  • No folder scanning or arbitrary non-HyperWiki project registration in v1.
  • No browser add-project flow in v1.
  • No shared cross-project terminal runtime.
