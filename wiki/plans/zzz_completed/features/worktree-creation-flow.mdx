<h1>Worktree Creation Flow</h1>
    <section className="summary">
      <h2>Summary</h2>
      <ul>
        <li>Status: complete</li>
        <li>Shape: compact feature plan</li>
        <li>Current unit: none</li>
        <li>Next action: monitor dogfooding feedback around duplicate branch handling and dependency install latency.</li>
        <li>Validation: <code>pnpm run check</code> and <code>pnpm run smoke:launch</code>.</li>
      </ul>
    </section>
    <section>
      <h2>Intent</h2>
      <p>Turn <code>+ worktree</code> into a first-class branch-switching flow. The topbar remains project-level, while the terminal branch rail changes from <code>main</code> to the worktree slug and reloads preserve the active checkout through the workspace URL.</p>
    </section>
    <section>
      <h2>Implementation</h2>
      <ul>
        <li>Add a compact terminal-toolbar popover with editable branch name, derived slug, path, and preview URL.</li>
        <li>Add a local API that creates a Git worktree, installs dependencies, registers it as a checkout, and returns the worktree workspace URL.</li>
        <li>After creation, navigate to the returned worktree URL with the current plan hash, reload project context from that checkout, and start the agent there.</li>
        <li>Keep checkout identity in the terminal branch rail with ellipsis overflow; do not add worktree state to the left topbar.</li>
      </ul>
    </section>
    <Verification>
      <ul>
        <li>Creating a worktree switches the URL and branch rail to the worktree slug.</li>
        <li>Reloading the worktree URL keeps the worktree active.</li>
        <li>The <code>+ worktree</code> affordance is hidden in non-main checkouts.</li>
        <li>Duplicate branch or path errors are shown inline without switching state.</li>
      </ul>
    </Verification>
    <section>
      <h2>Closeout</h2>
      <ul>
        <li>Implemented the branch-rail popover, create API, registry handoff, workspace URL switch, and existing-worktree agent prompt.</li>
        <li>Launch smoke coverage now creates a real worktree from the UI, verifies the URL and branch rail, reloads, and confirms the checkout remains active.</li>
        <li>Browser smoke coverage checks the popover preview behavior; the full browser smoke reached that flow and remains blocked later by the existing completed-stage dot color assertion.</li>
      </ul>
    </section>
    <section>
      <h2>Follow-up: Non-Git Projects</h2>
      <ul>
        <li>Status: complete.</li>
        <li>Intent: if a project has not been initialized as a Git repository, the branch rail should show <code>init git</code> instead of <code>+ worktree</code>.</li>
        <li>Implementation: reuse the existing Git initialization endpoint from the branch rail, refresh repo context after initialization, and only show the worktree creator once Git exists.</li>
        <li>Validation: <code>pnpm run check</code>, <code>pnpm run smoke:tauri-static-assets</code>, and a static browser check of the non-Git branch rail state.</li>
      </ul>
    </section>
