Reader Goal

After 2 minutes, a future agent should know which Hyperwiki scaffold conventions must be reconciled with the external Project HTML Wiki skill before editing generator code.

Intent

Hyperwiki already implements an HTML-first project memory layer, but its productized scaffold intentionally differs from the newly installed project-html-wiki skill in a few places. This plan made those differences explicit and updated generator code, prompts, and wiki guidance so future agents do not accidentally normalize away Hyperwiki-specific behavior.

Scope

  • Review src/init.js scaffold outputs against the skill's required project shape, naming, source index, planning, log, and automation-policy expectations.
  • Review new-project import handoff prompts in public/app.js so imported briefs ask agents to use the installed skill workflow without contradicting Hyperwiki conventions.
  • Update wiki guidance pages where needed to document intentional divergences: lowercase wiki/sources.html, generated wiki/AGENTS.html, app-served /assets/wiki.css, and Localhost Tooling runtime boundaries.
  • Record any unresolved skill-contract contradictions as explicit decisions or follow-ups instead of silently mixing conventions.

Non-Goals

  • Do not rename the existing source index to wiki/Sources.html unless a deliberate compatibility design is chosen.
  • Do not replace the app-served wiki CSS system with standalone embedded CSS across all generated pages without a product decision.
  • Do not create a repo-local maintainer skill unless the implementation decision says Hyperwiki should ship one for generated projects.
  • Do not rebootstrap this repository's existing wiki or rewrite completed plan history.

Acceptance

  • Generated Hyperwiki projects have one documented scaffold contract for source index naming, wiki agent guidance, styling portability, and Localhost Tooling boundaries.
  • Import handoff prompts tell agents when to apply the installed project-html-wiki workflow and when to preserve Hyperwiki product conventions.
  • Future agents can read this plan and avoid creating duplicate Sources.html, duplicate wiki guidance files, or unmanaged scaffold rewrites.
  • Checks and visual review confirm that scaffold changes do not break existing wiki navigation or new project creation flow.
