## Reader Goal

After 2 minutes, the reader can understand which richer planning components are being added, how future generated plans should use them, and how the change will be verified.

<Decision title="Direction">
  <p>Use Holocron as inspiration for a component cookbook and selected planning structures, not as a runtime dependency or docs-site shell.</p>
</Decision>

<PlanUnit title="Unit 01 - Component vocabulary and generation guidance">
  <p>Add <code>CardGroup</code>, <code>Columns</code>, <code>Column</code>, <code>Aside</code>, <code>RequestExample</code>, and <code>ResponseExample</code> to the local renderer and Rust wiki document pipeline.</p>
  <p>Update planning prompts, generated baseline guidance, and MDX artifact patterns so new and modified plans select richer structures when the content supports them.</p>
</PlanUnit>

<Evidence title="Source Basis">
  <ul>
    <li>Holocron uses component cookbook patterns such as cards, columns, examples, fields, steps, tabs, and visibility to make docs more scannable.</li>
    <li>hyperwiki already has a local component renderer, Markdown derivatives, and Rust component refs, so the lowest-risk path is to extend the existing vocabulary.</li>
  </ul>
</Evidence>
