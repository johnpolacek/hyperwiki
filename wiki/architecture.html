Architecture

hyperwiki is Localhost Tooling packaged as a single npm package with three surfaces: a CLI, repo-local canonical wiki files, and a localhost-only dev runtime.

Its trust boundary is the developer's own machine, repo files, Git state, terminal sessions, credentials, and environment variables. Hosted services are optional integrations, not the source of truth.

Boundaries

  • The CLI performs repeatable filesystem operations and starts the dev runtime.
  • The wiki is canonical repo-visible HTML.
  • Ideas are canonical repo-visible HTML under wiki/ideas/ until explicitly promoted into a registered hyperwiki project.
  • The dev runtime renders the wiki and terminal workspace but is not a hidden source of truth.
  • A user-level registry at ~/.hyperwiki/projects.json tracks known local hyperwiki projects for switching.
  • Runtime state is ignored under .hyperwiki/state/ and .hyperwiki/sessions/.
  • Multi-project runtime APIs resolve project roots through the registry, but each project's wiki, Git status, and terminal sessions remain isolated to that project root.
  • The workspace exposes local-only mode, canonical wiki/Git truth, ignored runtime state, session metadata, command history boundaries, and export/prune actions through the guardrails UI and /api/guardrails.
  • The Dashboard is a workspace route rather than a wiki file; it reads repo-local ideas and registered projects, then hands markdown-driven creation work to an agent terminal.
  • The Execute control is the primary start-work boundary: it creates terminals lazily and posts scoped main/worktree instructions into the visible agent session.
  • Current structured surfaces are local HTTP endpoints. A first-class verification-loop model, project contract schema, named review workflows, and MCP resources remain planned Stage 07 work.

Localhost Tooling Commitments

  • Bind local runtime surfaces to localhost addresses by default.
  • Keep canonical project knowledge in repo-visible files and Git.
  • Keep runtime state local and ignored unless a user intentionally exports or records it.
  • Expose permission, session, command-history, and export boundaries in the workspace instead of hiding them in app state.
  • Avoid default hosted sync, telemetry, remote execution, or cloud-backed knowledge stores.
